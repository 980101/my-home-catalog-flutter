# Harness Checklist

## 1. 문서 목적

이 문서는 `My Home Catalog` Android 앱을 Flutter로 마이그레이션할 때 기능 누락, UI 불일치, 예외 처리 누락, 플랫폼별 동작 차이를 검증하기 위한 체크리스트이다.

기준 문서:

- `docs/feature-spec.md`
- `docs/ui-guideline.md`
- `docs/coding-rules.md`

검증 원칙:

- `feature-spec.md`에 존재하는 기능만 검증한다.
- 확인되지 않은 기능을 기준으로 체크 항목을 만들지 않는다.
- Android 원본의 데이터 계약, 화면 이동, 예외 분기를 우선 유지한다.
- 구현 방식이 Android와 달라지는 경우 변경 이유와 검증 결과를 작업 로그 또는 PR 설명에 남긴다.
- 기능 검증은 코드 검증과 수동 동작 검증을 함께 수행한다.

## 2. 공통 완료 기준

모든 기능 단위는 다음 항목을 통과해야 완료로 본다.

- [ ] 구현 대상 기능이 `feature-spec.md`에 존재한다.
- [ ] Android 원본에 없는 화면, 버튼, 필터, 데이터 필드를 추가하지 않았다.
- [ ] 화면 이동 흐름이 `feature-spec.md`의 사용자 플로우와 일치한다.
- [ ] Firebase path, JSON key, 스타일 값, 가구 타입 값이 기존 계약과 일치한다.
- [ ] 비동기 상태가 `loading`, `data`, `empty`, `error`로 구분된다.
- [ ] 사용자에게 표시되는 메시지는 기존 문구를 우선 유지한다.
- [ ] 예외를 빈 `catch`로 무시하지 않는다.
- [ ] `build()` 메서드 안에서 네트워크 호출, 파일 IO, 상태 변경을 수행하지 않는다.
- [ ] `BuildContext`를 async gap 이후 사용할 때 `mounted`를 확인한다.
- [ ] `flutter analyze`에서 새 경고 또는 오류가 없다.
- [ ] 관련 단위 테스트, 위젯 테스트, 통합 테스트 또는 수동 검증 결과가 남아 있다.

## 3. 기능별 검증 체크리스트

### Initial 화면

대상 기능:

- 앱 실행 시 초기 화면 표시
- 추천 목록, 맞춤 가구 선택, 즐겨찾기 화면으로 이동

체크리스트:

- [ ] 앱 실행 시 초기 진입 화면이 열린다.
- [ ] 화면에 `My Home\n Catalog`가 표시된다.
- [ ] `맞춤 가구 둘러보기` 버튼이 표시된다.
- [ ] `바로 시작` 버튼이 표시된다.
- [ ] 즐겨찾기 이동 버튼이 표시된다.
- [ ] `바로 시작` 클릭 시 `MainPage`로 이동한다.
- [ ] `바로 시작`으로 이동한 `MainPage`는 기본값 `style=all`, `type=all`을 사용한다.
- [ ] `맞춤 가구 둘러보기` 클릭 시 `CustomPage`로 이동한다.
- [ ] 즐겨찾기 이동 버튼 클릭 시 `FavoritesPage`로 이동한다.
- [ ] 초기 화면에 원본 기능에 없는 검색, 로그인, 장바구니, 공유 기능이 없다.

### Main 추천 목록 화면

대상 기능:

- 전체 추천 목록 조회
- 스타일 필터와 가구 타입 필터 기반 조회
- 추천 아이템 상세 이동
- 하단 이동 버튼
- 뒤로가기 시 초기 화면 이동

체크리스트:

- [ ] Intent 또는 route parameter에 `style`이 없으면 `all`을 사용한다.
- [ ] Intent 또는 route parameter에 `type`이 없으면 `all`을 사용한다.
- [ ] 기본 제목은 `모든 스타일`이다.
- [ ] 스타일 필터가 `전체`, `내추럴`, `모던`, `클래식`, `젠`, `인더스트리얼`만 표시한다.
- [ ] 스타일 값은 각각 `all`, `natural`, `modern`, `classic`, `zen`, `industrial`로 매핑된다.
- [ ] 가구 타입 필터가 `All`, `Chair`, `Bed`, `Sofa`, `Dresser`, `Table`만 표시한다.
- [ ] 가구 타입 값은 각각 `all`, `chair`, `bed`, `sofa`, `dresser`, `table`로 매핑된다.
- [ ] 스타일 필터 선택 시 선택된 버튼의 배경과 텍스트 색상이 변경된다.
- [ ] 스타일 필터 선택 시 가구 타입이 `all`로 초기화된다.
- [ ] 스타일 필터 변경 시 기존 추천 목록 데이터가 비워지고 새 기준으로 조회된다.
- [ ] 가구 타입 필터 선택 시 선택된 버튼의 배경이 변경된다.
- [ ] 가구 타입 필터 변경 시 기존 추천 목록 데이터가 비워지고 새 기준으로 조회된다.
- [ ] 추천 목록은 세로 리스트로 표시된다.
- [ ] 추천 아이템은 `image`, `name`, `price`를 표시한다.
- [ ] 추천 아이템 클릭 시 `DetailPage`로 이동한다.
- [ ] `DetailPage`로 `image`, `name`, `price`, `link` 값이 전달된다.
- [ ] 하단 버튼으로 `CustomPage`, `InitialPage`, `FavoritesPage`로 이동할 수 있다.
- [ ] Android 뒤로가기 또는 이에 대응하는 뒤로가기 동작이 `InitialPage` 이동 정책과 충돌하지 않는다.

### Custom 가구 선택 화면

대상 기능:

- 가구 타입 선택
- 선택한 가구 타입을 카메라 화면으로 전달
- 미선택 예외 처리

체크리스트:

- [ ] 화면에 `맞춤 가구 둘러보기`가 표시된다.
- [ ] 화면에 `가구 선택`이 표시된다.
- [ ] 화면에 `다음` 버튼이 표시된다.
- [ ] 가구 타입이 2열 그리드로 표시된다.
- [ ] 표시 값은 `all`, `chair`, `bed`, `sofa`, `dresser`, `table`만 포함한다.
- [ ] 각 가구 타입 아이템은 아이콘과 텍스트를 포함한다.
- [ ] 선택된 가구 타입 아이템의 배경 또는 border가 변경된다.
- [ ] 가구 타입을 선택하면 선택 값이 저장된다.
- [ ] 선택 후 `다음` 클릭 시 `CameraPage`로 이동한다.
- [ ] `CameraPage`에 선택된 `type` 값이 전달된다.
- [ ] 선택 없이 `다음` 클릭 시 `"가구를 선택해주세요!"` 메시지가 표시된다.
- [ ] 선택 없이 `다음` 클릭 시 `CameraPage`로 이동하지 않는다.

### Camera 및 TFLite 분류 화면

대상 기능:

- 카메라 권한 확인 및 요청
- 카메라 프리뷰 표시
- 카메라 프레임 분류
- 인식 스타일 저장
- 히스토리 표시, 재탐색, 삭제
- 촬영 후 추천 목록 이동

체크리스트:

- [ ] `CameraPage` 진입 시 전달받은 `type` 값을 유지한다.
- [ ] 카메라 권한이 없으면 권한 요청 흐름이 실행된다.
- [ ] 권한 설명이 필요한 경우 `"Camera permission is required for this demo"` 메시지가 표시된다.
- [ ] 권한 승인 후 카메라 프리뷰가 표시된다.
- [ ] 권한 거부 시 카메라 프리뷰가 표시되지 않으며 앱이 크래시하지 않는다.
- [ ] 화면에 `촬영하기` 버튼이 표시된다.
- [ ] TFLite 모델 파일 `model.tflite`가 asset으로 등록되어 있다.
- [ ] 라벨 파일 `labels.txt`가 asset으로 등록되어 있다.
- [ ] 라벨 값은 `zen`, `natural`, `modern`, `industrial`, `classic`만 사용한다.
- [ ] 카메라 프레임은 회전 보정, 정사각형 crop/pad, resize 처리 기준을 따른다.
- [ ] 추론은 UI thread를 막지 않는다.
- [ ] 처리 중인 프레임이 있으면 추가 프레임을 처리하지 않는다.
- [ ] 최상위 결과 confidence가 `0.9` 이상일 때만 `recognitionStyle`을 갱신한다.
- [ ] confidence가 `0.9` 미만이면 `recognitionStyle`을 갱신하지 않는다.
- [ ] 상위 3개 결과를 confidence 내림차순으로 다루는 계약을 유지한다.
- [ ] `recognitionStyle`이 비어 있으면 `촬영하기` 클릭 시 `MainPage`로 이동하지 않는다.
- [ ] `recognitionStyle`이 있으면 `촬영하기` 클릭 시 히스토리에 저장된다.
- [ ] `recognitionStyle`이 있으면 `MainPage`로 `style`과 `type`이 전달된다.
- [ ] 최근 인식 스타일 히스토리가 가로 목록으로 표시된다.
- [ ] 히스토리 목록은 저장소의 스타일 값을 읽어 표시한다.
- [ ] 히스토리 아이템 클릭 시 `MainPage`로 이동하고 해당 `style`이 전달된다.
- [ ] 히스토리 아이템 클릭 시 현재 가구 타입 `type`이 함께 전달된다.
- [ ] 히스토리 삭제 버튼 클릭 시 화면 목록에서 제거된다.
- [ ] 히스토리 삭제 버튼 클릭 시 저장소에서도 해당 style key가 제거된다.
- [ ] 카메라 이미지가 없거나 프리뷰 크기가 준비되지 않은 경우 처리를 중단하고 크래시하지 않는다.

### Detail 상품 상세 화면

대상 기능:

- 추천/즐겨찾기 아이템 상세 표시
- 구매 링크 외부 앱 열기
- 즐겨찾기 저장 및 중복 처리

체크리스트:

- [ ] `DetailPage`는 전달받은 `image`, `name`, `price`, `link`만 사용한다.
- [ ] 화면에 `제품명` 라벨이 표시된다.
- [ ] 화면에 상품명이 표시된다.
- [ ] 화면에 `가격` 라벨이 표시된다.
- [ ] 화면에 가격이 표시된다.
- [ ] 상품 이미지는 전달받은 `image` 값으로 로드된다.
- [ ] 화면에 `구매하기` 버튼이 표시된다.
- [ ] `구매하기` 클릭 시 `link`가 외부 브라우저 또는 외부 앱으로 열린다.
- [ ] 앱 내부 WebView를 추가하지 않았다.
- [ ] 즐겨찾기 저장 여부를 확인한다.
- [ ] 이미 저장된 항목이면 저장 버튼이 채워진 상태로 표시된다.
- [ ] 저장 버튼 클릭 시 현재 물품이 `savedItem.json`에 저장된다.
- [ ] 저장 필드는 `Image`, `Name`, `Price`, `Link` 계약을 따른다.
- [ ] 중복 체크 기준은 `Name`이다.
- [ ] 같은 `Name`의 항목이 이미 있으면 새 항목을 추가하지 않는다.
- [ ] 중복 저장 시 `"이미 존재하는 아이템입니다."` 메시지가 표시된다.
- [ ] JSON 객체 생성 실패 시 오류가 로깅되거나 error 상태로 전환된다.

### Favorites 즐겨찾기 화면

대상 기능:

- 즐겨찾기 목록 조회
- 즐겨찾기 상세 이동
- 삭제 대상 선택
- 선택 항목 삭제

체크리스트:

- [ ] 화면에 `즐겨찾기` 제목이 표시된다.
- [ ] `savedItem.json`을 읽어 즐겨찾기 목록을 구성한다.
- [ ] `savedItem.json` 파일이 없을 때 앱이 크래시하지 않는다.
- [ ] JSON 배열의 `Image`, `Name`, `Price`, `Link` 값을 `ItemData` 대응 모델로 변환한다.
- [ ] 즐겨찾기 목록은 세로 리스트로 표시된다.
- [ ] 즐겨찾기 아이템은 이미지, 이름, 가격을 표시한다.
- [ ] 즐겨찾기 아이템 클릭 시 `DetailPage`로 이동한다.
- [ ] `DetailPage`로 `image`, `name`, `price`, `link` 값이 전달된다.
- [ ] 즐겨찾기 아이템에 `선택` 버튼이 표시된다.
- [ ] `선택` 버튼 클릭 시 삭제 대상 position이 저장된다.
- [ ] 상단 삭제 버튼 클릭 시 선택된 position의 항목이 삭제된다.
- [ ] 삭제 후 `savedItem.json`이 갱신된다.
- [ ] 삭제 후 화면 목록이 갱신된다.
- [ ] 목록이 비어 있거나 삭제 대상이 선택되지 않은 상태에서 삭제 버튼 클릭 시 `"즐겨찾기한 항목이 없습니다 !"` 메시지가 표시된다.
- [ ] JSON 파싱 실패 시 오류가 로깅되거나 error 상태로 전환된다.

## 4. 데이터 계약 검증 체크리스트

### 공통 모델

- [ ] 추천/즐겨찾기 공통 모델은 `image`, `name`, `price`, `link` 필드를 유지한다.
- [ ] Firebase 응답 필드명은 `image`, `name`, `price`, `link` 소문자 key를 사용한다.
- [ ] 로컬 즐겨찾기 JSON key가 Firebase key와 다를 경우 mapper가 분리되어 있다.
- [ ] UI 표시용 문자열 변환이 모델 내부에 과도하게 들어가지 않았다.

### 스타일 값

- [ ] 앱 내부 스타일 값은 `all`, `natural`, `modern`, `classic`, `industrial`, `zen` 범위만 사용한다.
- [ ] TFLite 라벨 값은 `zen`, `natural`, `modern`, `industrial`, `classic` 범위만 사용한다.
- [ ] Firebase path에 표시용 한글 텍스트가 들어가지 않는다.

### 가구 타입 값

- [ ] 앱 내부 가구 타입 값은 `all`, `bed`, `chair`, `dresser`, `sofa`, `table` 범위만 사용한다.
- [ ] Firebase path에 표시용 영문 대문자 텍스트가 들어가지 않는다.
- [ ] `All`, `Chair`, `Bed`, `Sofa`, `Dresser`, `Table`은 화면 표시 텍스트로만 사용된다.

### 화면 간 전달 값

- [ ] `CustomPage`에서 `CameraPage`로 `type`이 전달된다.
- [ ] `CameraPage`에서 `MainPage`로 `style`과 `type`이 전달된다.
- [ ] 히스토리 아이템에서 `MainPage`로 `style`과 `type`이 전달된다.
- [ ] 추천 목록에서 `DetailPage`로 `image`, `name`, `price`, `link`가 전달된다.
- [ ] 즐겨찾기 목록에서 `DetailPage`로 `image`, `name`, `price`, `link`가 전달된다.

## 5. UI 검증 체크리스트

### 공통 UI

- [ ] 색상, 간격, radius, 텍스트 스타일이 토큰으로 관리된다.
- [ ] 화면에서 색상과 간격을 반복적으로 하드코딩하지 않는다.
- [ ] 기본 배경은 `background`, 카드와 리스트 아이템은 `surface` 계열을 사용한다.
- [ ] 주요 액션과 선택 상태에 `primary` 계열을 사용한다.
- [ ] 터치 가능한 영역은 최소 `48 x 48`을 확보한다.
- [ ] `SafeArea`를 사용해 상태바, 노치, 홈 인디케이터 영역을 침범하지 않는다.
- [ ] 화면 제목은 한 화면에 하나만 둔다.
- [ ] 상품명은 최대 2줄 또는 overflow 처리를 가진다.
- [ ] 가격은 원본 `price` 문자열을 임의 포맷 변경 없이 표시한다.

### 버튼

- [ ] Primary button은 핵심 액션에만 사용한다.
- [ ] Primary button 높이는 `52` 기준을 따른다.
- [ ] Secondary button은 보조 이동 또는 `선택` 버튼에 사용된다.
- [ ] Icon button은 최소 터치 영역과 접근성 label 또는 tooltip을 가진다.
- [ ] 즐겨찾기 저장 버튼은 저장 여부를 outline/fill 또는 색상으로 구분한다.
- [ ] 삭제 버튼은 삭제 동작임을 구분할 수 있는 시각 상태를 가진다.

### 필터와 선택 상태

- [ ] 스타일 필터의 선택/비선택 상태가 명확하다.
- [ ] 가구 타입 필터의 선택/비선택 상태가 명확하다.
- [ ] 선택 상태는 색상만이 아니라 배경, border, 텍스트 굵기 중 하나 이상으로 함께 구분된다.
- [ ] 필터 영역은 작은 화면에서 overflow 없이 가로 스크롤 또는 줄바꿈으로 표시된다.

### 카드와 리스트

- [ ] 추천 상품 카드가 이미지, 상품명, 가격을 포함한다.
- [ ] 즐겨찾기 상품 카드가 이미지, 상품명, 가격, `선택` 버튼을 포함한다.
- [ ] 카드 전체 클릭 영역이 상세 이동과 연결된다.
- [ ] 이미지 로딩 실패 시 placeholder 영역이 표시된다.
- [ ] 이미지가 카드 radius 밖으로 넘치지 않는다.
- [ ] 리스트 마지막 아이템이 하단 버튼 또는 safe area에 가리지 않는다.
- [ ] 히스토리 리스트는 가로 스크롤로 표시된다.
- [ ] 히스토리 아이템의 이동 영역과 삭제 버튼 영역이 구분된다.

### 상태 화면

- [ ] 로딩 상태는 `CircularProgressIndicator` 또는 동일한 로딩 표시를 사용한다.
- [ ] 빈 추천 목록은 빈 상태 UI로 표시된다.
- [ ] 빈 즐겨찾기 목록은 빈 상태 UI로 표시된다.
- [ ] 빈 히스토리 목록은 빈 상태 UI로 표시된다.
- [ ] 에러 상태는 error 색상 또는 명확한 오류 표시를 사용한다.
- [ ] 재시도 버튼은 실제 재시도 동작이 구현된 경우에만 표시된다.

## 6. 예외 처리 검증 체크리스트

### 사용자 메시지

- [ ] 가구 미선택 상태에서 `다음` 클릭 시 `"가구를 선택해주세요!"`가 표시된다.
- [ ] 이미 저장된 상품 저장 시 `"이미 존재하는 아이템입니다."`가 표시된다.
- [ ] 즐겨찾기 목록이 비어 있거나 삭제 대상이 없을 때 삭제 클릭 시 `"즐겨찾기한 항목이 없습니다 !"`가 표시된다.
- [ ] 카메라 권한 설명 필요 시 `"Camera permission is required for this demo"`가 표시된다.
- [ ] Camera2 또는 카메라 캡처 세션 설정 실패에 대응하는 `"Failed"` 메시지 또는 로깅 정책이 있다.

### 로그 또는 error 상태

- [ ] 즐겨찾기 JSON 읽기 실패가 기록되거나 error 상태로 전환된다.
- [ ] 즐겨찾기 JSON 쓰기 실패가 기록되거나 error 상태로 전환된다.
- [ ] 즐겨찾기 JSON 파싱 실패가 기록되거나 error 상태로 전환된다.
- [ ] 즐겨찾기 삭제 처리 실패가 기록되거나 error 상태로 전환된다.
- [ ] Firebase 조회 취소 또는 실패가 기록되거나 error 상태로 전환된다.
- [ ] 카메라 접근 실패가 기록되거나 error 상태로 전환된다.
- [ ] 카메라 프레임 처리 실패가 기록되거나 error 상태로 전환된다.
- [ ] classifier 생성 실패가 기록되거나 error 상태로 전환된다.
- [ ] TFLite 추론 실패가 기록되거나 error 상태로 전환된다.
- [ ] 외부 링크 실행 실패가 사용자 안내 또는 로깅 정책에 따라 처리된다.

### 조용히 중단되는 분기

- [ ] 인식 스타일이 비어 있으면 추천 화면으로 이동하지 않는다.
- [ ] 이미 처리 중인 카메라 프레임이 있으면 새 프레임을 버린다.
- [ ] Camera2 이미지가 없거나 프리뷰 크기가 준비되지 않았으면 프레임 처리를 중단한다.
- [ ] classifier 생성 실패 시 이후 초기화 또는 추론이 크래시 없이 중단된다.
- [ ] `savedItem.json` 파일이 없으면 빈 상태 또는 파일 없음 상태로 처리된다.
- [ ] 기존 즐겨찾기 저장 데이터가 없으면 중복 체크 결과가 false로 처리된다.

## 7. Firebase 연동 검증 체크리스트

### 설정과 계층

- [ ] Firebase Realtime Database 조회 기능은 `MainPage`에 직접 작성되지 않는다.
- [ ] Firebase 접근은 data source 또는 repository 계층으로 분리되어 있다.
- [ ] Firebase path와 key는 상수화되어 있다.
- [ ] Firebase Analytics 이벤트를 임의로 추가하지 않았다.

### 조회 경로

- [ ] 기본 참조는 `all`이다.
- [ ] 실제 조회 경로는 `all/{style}/{type}`이다.
- [ ] `style=all`이면 `natural`, `modern`, `classic`, `industrial`, `zen` 전체를 순회한다.
- [ ] `type=all`이면 `bed`, `chair`, `dresser`, `sofa`, `table` 전체를 순회한다.
- [ ] 개별 style 값이면 해당 style만 조회한다.
- [ ] 개별 type 값이면 해당 type만 조회한다.

### 조회 결과

- [ ] 각 leaf child를 `image`, `name`, `price`, `link` 필드 모델로 변환한다.
- [ ] 조회 결과가 추천 목록에 반영된다.
- [ ] 스타일 변경 후 이전 목록이 남아 있지 않다.
- [ ] 타입 변경 후 이전 목록이 남아 있지 않다.
- [ ] Firebase 조회 실패 또는 취소가 처리된다.

## 8. 로컬 저장 검증 체크리스트

### 즐겨찾기 JSON

- [ ] 파일명은 `savedItem.json` 계약을 따른다.
- [ ] 저장 형식은 JSON 배열이다.
- [ ] 저장 key는 `Image`, `Name`, `Price`, `Link` 계약을 따른다.
- [ ] 저장 전 `Name` 기준으로 중복 여부를 확인한다.
- [ ] 중복 항목은 JSON 배열에 추가하지 않는다.
- [ ] 조회 시 JSON 배열을 즐겨찾기 목록으로 변환한다.
- [ ] 삭제 시 선택된 position을 제외한 JSON 배열로 다시 저장한다.
- [ ] 삭제 후 화면 상태가 즉시 갱신된다.

### 스타일 히스토리

- [ ] 저장 key는 인식 스타일 문자열이다.
- [ ] 저장 value는 인식 스타일 문자열이다.
- [ ] 히스토리는 `CameraPage` 진입 시 저장소에서 읽어온다.
- [ ] 동일 스타일 저장 시 기존 값을 제거 후 다시 저장하는 원본 동작을 유지하거나, 변경 시 이유를 기록한다.
- [ ] 히스토리 삭제 시 화면 목록과 저장소가 함께 갱신된다.

## 9. Android/iOS 플랫폼 검증 체크리스트

### 공통 동작

- [ ] Android와 iOS에서 동일한 화면 플로우를 제공한다.
- [ ] Android와 iOS에서 동일한 스타일 값과 가구 타입 값을 사용한다.
- [ ] Android와 iOS에서 추천 목록, 상세, 즐겨찾기 기능의 데이터 계약이 동일하다.
- [ ] Android와 iOS에서 외부 링크가 외부 브라우저 또는 외부 앱으로 열린다.
- [ ] Android와 iOS에서 앱 내부 WebView가 사용되지 않는다.

### 권한과 카메라

- [ ] Android에서 카메라 권한 요청이 동작한다.
- [ ] iOS에서 카메라 권한 요청이 동작한다.
- [ ] 권한 거부 시 Android와 iOS 모두 크래시하지 않는다.
- [ ] Android에서 카메라 프리뷰가 표시된다.
- [ ] iOS에서 카메라 프리뷰가 표시된다.
- [ ] Android와 iOS에서 TFLite 모델과 라벨 asset을 읽을 수 있다.
- [ ] Android와 iOS에서 confidence `0.9` 기준이 동일하게 적용된다.

### 화면 안전 영역

- [ ] Android status bar와 navigation bar에 콘텐츠가 가리지 않는다.
- [ ] iOS notch와 home indicator에 콘텐츠가 가리지 않는다.
- [ ] 하단 버튼과 리스트 마지막 아이템이 플랫폼별 safe area에 가리지 않는다.
- [ ] iOS back gesture와 Android back button 동작이 라우팅 정책과 충돌하지 않는다.

## 10. 성능 및 사용성 검증 체크리스트

### 성능

- [ ] Firebase 조회는 화면 build 중 반복 실행되지 않는다.
- [ ] 파일 IO는 화면 build 중 실행되지 않는다.
- [ ] TFLite 추론은 UI thread를 장시간 막지 않는다.
- [ ] 카메라 프레임 처리 중 중복 프레임을 처리하지 않는다.
- [ ] 추천 목록과 즐겨찾기 목록은 스크롤 가능한 리스트로 렌더링된다.
- [ ] 네트워크 이미지 로딩 중 UI가 멈추지 않는다.
- [ ] 이미지 로딩 실패가 리스트 전체 렌더링 실패로 이어지지 않는다.

### 사용성

- [ ] 주요 버튼은 한 손 조작 가능한 크기와 간격을 가진다.
- [ ] 필터 선택 상태를 즉시 확인할 수 있다.
- [ ] 가구 타입 선택 상태를 즉시 확인할 수 있다.
- [ ] 상품명, 가격, 버튼 텍스트가 작은 화면에서 겹치지 않는다.
- [ ] 상품명은 길어도 레이아웃을 깨지 않는다.
- [ ] 카메라 화면에서 프리뷰, 히스토리, `촬영하기` 버튼이 서로 가리지 않는다.
- [ ] 빈 목록, 로딩, 에러 상태에서 기존 화면 이동 경로가 사라지지 않는다.

## 11. AI 생성 코드 검증 체크리스트

코드 생성 또는 수정 후 AI는 다음 항목을 확인한다.

- [ ] 구현 대상이 `feature-spec.md`에 있는 기능이다.
- [ ] `ui-guideline.md`의 색상, 텍스트, 버튼, 카드, 상태 화면 규칙을 따른다.
- [ ] `coding-rules.md`의 Feature First 구조와 책임 분리 규칙을 따른다.
- [ ] 상태관리, 라우팅, 저장소, 패키지 선택을 임의로 확정하지 않았다.
- [ ] Firebase, 파일 저장, TFLite 추론 로직이 화면 Widget에 직접 들어가지 않았다.
- [ ] 공통 컴포넌트가 특정 feature의 비즈니스 로직에 의존하지 않는다.
- [ ] 신규 기능, 신규 데이터 필드, 신규 Firebase 경로를 추가하지 않았다.
- [ ] 기존 Android 동작과 달라지는 부분이 있으면 이유를 기록했다.
- [ ] 테스트 또는 수동 검증 결과를 작업 결과에 남겼다.
